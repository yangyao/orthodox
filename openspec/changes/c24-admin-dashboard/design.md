## Context

当前后台已经具备壳应用、认证能力以及若干业务管理页，但管理员登录后还缺少一个“全局态势”页面。实施计划中的 C24 明确要求提供三组接口：

- `GET /api/admin/v1/dashboard/overview`
- `GET /api/admin/v1/dashboard/learning`
- `GET /api/admin/v1/dashboard/products`

这些接口的数据来源横跨交易域和学习域：订单、支付、钱包充值、练习会话、模考会话都需要参与聚合。由于现阶段系统仍运行在单体 Next.js + PostgreSQL 模式下，本 change 的核心是定义清晰的统计口径和可维护的聚合查询，而不是引入新的 BI 基础设施。

## Goals / Non-Goals

**Goals:**
- 为后台首页提供统一的总览、学习、商品三类聚合接口。
- 明确定义首版指标口径，包括 GMV、付费用户、活跃学习用户、销量等。
- 提供后台首页卡片、图表和榜单所需的数据结构。
- 保持接口受现有后台认证保护，避免暴露内部经营数据。

**Non-Goals:**
- 不构建独立数据仓库、ETL 或离线报表体系。
- 不支持任意拖拽自定义看板或用户自定义时间窗口。
- 不处理财务对账级别的精确结算逻辑；首版看板以运营视角为主。
- 不覆盖 C23 的小程序用户端学习统计页面。

## Decisions

### 1. 看板按业务域拆成三个接口，而不是一个超大聚合接口

- `overview` 返回首页第一屏核心 KPI 和近 7/30 天趋势摘要。
- `learning` 专注练习/模考活跃与学习趋势。
- `products` 专注商品销量、充值表现和支付方式构成。

**为什么这样做:**
- 后台首页虽然是一个页面，但不同区块刷新频率和数据来源不同。
- 一个超大接口会让口径、缓存和查询复杂度全部耦合。

### 2. GMV 与充值金额分开统计，避免经营数据混淆

- `overview` 中的 GMV 仅统计商品订单支付金额（题库/资料包）。
- 钱包充值金额单独作为 `rechargeAmountFen` 暴露在 `products` 或 `overview` 的独立字段中。

**为什么这样做:**
- 充值不是最终商品消费，如果直接并入 GMV，会放大经营规模。
- 原型和实施计划里同时存在商品交易和钱包充值，这两者必须口径分离。

### 3. 学习活跃采用“有学习行为的去重用户数”口径

- `learning` 接口中的活跃学习用户，定义为在指定统计周期内至少完成一次练习会话或提交一次模考的去重用户数。
- 做题量优先使用完成的练习会话和已保存答案数聚合，模考量优先使用已提交模考试卷数聚合。

**为什么这样做:**
- 用户登录或浏览首页不应算作学习活跃。
- 只用会话或只用答案都不完整，首版按学习行为去重更接近真实活跃。

### 4. 商品榜单基于 SKU/商品聚合，但对外仍以管理员可读名称返回

- `products` 接口按 `catalog_products` / `product_skus` 关联 `order_items` 聚合销量、销售额。
- 返回时同时提供 `productId`、`skuId`、`title`、`productType`、`soldCount`、`paidAmountFen`。

**为什么这样做:**
- 后台需要看到实际卖得好的商品和 SKU，而不是只有订单数字。
- 对前端直接返回可读名称，避免再做一轮客户端关联。

### 5. 首版采用固定时间窗口与有限筛选参数

- 三个接口都支持 `range=7d|30d`，必要时支持 `bankId` 或 `productType` 等有限筛选。
- 不支持任意日期区间输入。

**为什么这样做:**
- 原型只需要首页运营看板，不是完整 BI 工具。
- 固定窗口可以简化 SQL 和缓存策略，也更利于后续做接口级缓存。

### 6. 先用在线聚合查询，后续再考虑预聚合

- 聚合来源：
  - `orders`, `order_items`, `payment_transactions`, `wallet_ledger_entries`
  - `practice_sessions`, `practice_answers`
  - `exam_sessions`
- 在数据库层面补齐 `(status, paid_at)`、`(user_id, created_at)`、`(product_id, sku_id)` 等索引。

**为什么这样做:**
- 当前阶段数据规模较小，在线聚合可维护性更高。
- 如果未来后台看板访问频率或数据量上升，再单独抽出预聚合 change 更合理。

## Risks / Trade-offs

- **[Risk] 多表聚合 SQL 复杂且易出错** → 通过共享的 dashboard aggregation helper 收敛口径，并为每个指标写用例。
- **[Risk] GMV、充值、余额支付口径被混淆** → 在接口字段命名上显式拆开 `gmv`、`rechargeAmount`、`walletPaidAmount`。
- **[Risk] 在线聚合在数据量增长后变慢** → 先补索引和固定时间窗口，后续需要时再增加缓存或预聚合。
- **[Risk] 首页一次性请求太多数据** → 拆成 3 个接口，前端可并行请求并独立处理加载态。

## Migration Plan

1. 检查交易域和学习域表结构是否已具备统计所需时间字段和状态字段。
2. 为后台看板查询补充必要索引。
3. 上线 `/api/admin/v1/dashboard/*` 三组接口。
4. 实现后台首页卡片、图表和榜单组件。

回滚策略：
- 若某个区块统计口径异常，可单独下线对应接口或隐藏对应卡片。
- 若首页图表渲染异常，可保留总览卡片，临时移除趋势图，不影响后台其他管理功能。

## Open Questions

- 商品榜单是否需要区分题库与资料包两个榜单；首版建议先支持总榜并提供 `productType` 字段。
- 后台首页默认时间窗口是 7 天还是 30 天；首版建议默认 7 天，并允许切换到 30 天。
