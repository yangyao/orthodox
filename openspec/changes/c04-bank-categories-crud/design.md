## Context

P0 已完成：管理后台骨架（C01）、数据库连接（C02）、管理员认证（C03）。现在开始 P1 题库内容域。本 Change 建立最上层的分类和题库实体，后续卷册/章节/题目都依赖题库 ID。

## Goals / Non-Goals

**Goals:**

- 管理员可以创建/编辑/删除题库分类
- 管理员可以在分类下创建/编辑/归档题库
- 分类支持排序和显示/隐藏控制
- 题库列表支持分页和状态筛选
- API 使用 Zod 做请求参数校验

**Non-Goals:**

- 不实现卷册/章节管理（属于 C05）
- 不实现题目管理（属于 C06）
- 不实现题库封面图片上传（暂用 URL 输入）
- 不实现前台小程序端的题库展示 API（属于 C09）

## Decisions

### 1. API 路由组织

**选择**: 所有 admin API 放在 `src/app/api/admin/v1/` 下，使用 Next.js Route Handlers

**理由**:
- 与架构文档 API 设计一致（基础路径 `/api/admin/v1`）
- Route Handlers 天然支持 RESTful 风格
- 每个 Route Handler 文件可以独立做权限校验

### 2. Zod 请求校验

**选择**: 使用 Zod schema 校验所有 API 请求体

**理由**:
- 类型安全：从 Zod schema 推导 TypeScript 类型
- 统一错误响应格式
- 与架构文档统一响应格式对齐

### 3. 管理页面模式

**选择**: 列表页用 Server Component + 客户端表格，新增/编辑用独立页面（非弹窗）

**理由**:
- 题库编辑表单字段较多（名称、副标题、分类、封面、描述、排序、推荐、售卖类型等），弹窗空间不够
- 独立页面 URL 可分享，刷新不丢失
- 分类字段少，用弹窗即可

### 4. 表格组件

**选择**: 直接用 shadcn/ui 的 Table 组件 + 手写分页

**理由**:
- 初期不需要 TanStack Table 的复杂功能（排序、虚拟滚动等）
- 保持依赖最小化，后续需要时再引入

## Risks / Trade-offs

- **[表结构变更]** `question_banks` 表后续 C16 需要关联商品，但本 Change 先建基础字段，不预留商品相关字段 → 后续通过 migration 添加关联字段即可，不影响现有数据。
- **[无图片上传]** 封面 URL 暂时只支持手动输入外部链接 → 后续可接入腾讯云 COS 图片上传。
